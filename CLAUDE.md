# rust_formatter

A Rust code formatter that respects user intent. It normalizes horizontal spacing,
indentation, and trailing whitespace. It also reformats method chains that exceed
the max line length (120 chars).

## Build & Test
```
cargo build
cargo test
cargo clippy
```

## Architecture
- `src/main.rs` - CLI entry point (clap). Supports stdin, `--write`, `--check`.
- `src/lib.rs` - Re-exports `formatter` module.
- `src/formatter.rs` - Core formatting logic using `ra_ap_syntax` lossless CST.
- `tests/formatter_tests.rs` - Fixture-based integration tests.
- `tests/fixtures/` - Test fixtures as `.input.rs` / `.expected.rs` pairs.

## Adding Tests
Drop files in `tests/fixtures/`:
- `my_test.input.rs` - the input source code
- `my_test.expected.rs` - the expected formatted output (omit if input = expected)

Then register the test name in the `fixture_tests!` macro in `tests/formatter_tests.rs`.

## WIP Workflow
`tests/wip/` contains scratch files for iterating on formatting behavior:
- `input.rs` - source code to format (user writes this)
- `output.rs` - formatter output (generated by `cat tests/wip/input.rs | cargo run > tests/wip/output.rs`)
- `expected.rs` - desired output (user writes this to show what the result should look like)

Workflow: user edits `input.rs` and `expected.rs`, then asks to run the formatter
and diff `output.rs` vs `expected.rs` to identify what needs fixing.

**IMPORTANT**: Do NOT modify or delete existing tests unless the user explicitly says
to do so. If a proposed change would break existing tests, ask the user what to do
before proceeding. When asking, implement the change first, run the tests, and show
the user the affected tests with their diffs (old expected vs new actual output).

**IMPORTANT**: After implementing any formatting rule or fix, ALWAYS add a test for
it in `tests/fixtures/` and register it in the `fixture_tests!` macro. No exception.

## Key Design Decisions
- Uses `ra_ap_syntax` (rust-analyzer's parser) for lossless CST parsing.
- Iterates tokens in document order, applying spacing rules between adjacent
  non-whitespace tokens.
- Line breaks are generally preserved; only horizontal spacing is modified.
- Method chain reformatting (max line length = 120):
  - If a chain fits on one line, it's collapsed to a single line.
  - If a chain exceeds 120 chars and has no existing breaks, all method-call dots
    are broken to separate lines. Field-access dots are NOT auto-broken.
  - If a chain exceeds 120 chars and has existing breaks, the first-line grouping
    (dots before the first break) is preserved, and all method-call dots after the
    first break are expanded to separate lines.
  - Chains nested inside other chains' arguments are skipped (handled by the outer
    chain's reformatting).
  - Pipeline: format_whitespace → reformat_chains → format_whitespace (two passes
    to ensure correct indentation after chain line changes).
- Indentation is computed from the CST tree structure (not preserved from source).
  Indent-increasing nodes: STMT_LIST, ITEM_LIST, ASSOC_ITEM_LIST, MATCH_ARM_LIST,
  RECORD_FIELD_LIST, RECORD_EXPR_FIELD_LIST, RECORD_PAT_FIELD_LIST, VARIANT_LIST,
  USE_TREE_LIST, EXTERN_ITEM_LIST, and TOKEN_TREE (only `{}` and `[]` delimited).
  Delimiter tokens ({, }, [, ]) stay at the parent indent level.
- Continuation indent: lines starting with `.` (method chains) get +1 indent level.
- Multiline string literals: content is re-indented to match surrounding code's
  indent level (content at indent+1, closing quote at indent). Relative indentation
  within strings is preserved.
- Derive args sorted alphabetically: `#[derive(Debug, Clone)]` → `#[derive(Clone, Debug)]`.
- Imports: flattened to one-per-line, sorted alphabetically, grouped with blank line
  separators. Group order: mod declarations → star imports → foreign → crate → pub re-exports.
- Trailing whitespace removed, trailing newline ensured.
